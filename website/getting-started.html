Getting Started

Start Paperwise with Docker or from source.

Two setup paths: run the full self-hosted stack with Docker Compose (recommended), or work from a local checkout in development mode.

Recommended

Docker Compose

Best for most people. Runs the API, worker, Redis, and Postgres as one self-hosted stack with persistent volumes.

  • Fastest path to a working install
  • Matches the repo’s intended deployment flow
  • Uses persistent Docker volumes for app data
Use Docker Open repo

Developer

Local Checkout

Best if you want to modify the code, run the backend and worker from a local virtualenv, or iterate on the app in development mode.

  • Runs the app directly from source
  • Uses `make setup` and `make dev-up` from the repo
  • Good for debugging, hacking, and UI/API changes
Run locally Read support

Recommended

Option 1: Docker container stack

This uses the repository’s root `docker compose` stack, which includes `api`, `worker`, `redis`, and `postgres`.

Prerequisites Docker Engine or Docker Desktop with Compose support.
Port The Docker stack publishes the web UI on http://localhost:8080.

1. Clone the repository

git clone https://github.com/zellux/paperwise.git
cd paperwise

2. Create a deployment env file

cp .env.docker.example .env

Set a strong auth secret before starting the stack. 

PAPERWISE_AUTH_SECRET=replace-with-a-strong-secret

3. Build and start the stack

docker compose up -d --build

4. Open Paperwise

http://localhost:8080
Command What it does
docker compose ps Show service status for the self-hosted stack.
docker compose logs -f api worker Follow backend and worker logs.
docker compose down Stop the stack.

Developer

Option 2: Local code checkout and dev mode

This path uses the repo’s Make targets. It is the better option if you want to work on the codebase or run the app directly from source.

Prerequisites Python 3.13 recommended, plus Docker available for local dependency services.
Port The local development UI runs on http://localhost:8000.

1. Clone the repository

git clone https://github.com/zellux/paperwise.git
cd paperwise

2. Copy the local env file

cp .env.example .env.local

3. Create the virtualenv and install dependencies

make setup

If needed, you can explicitly choose Python 3.13: 

make setup PYTHON=python3.13

4. Start local development services

make dev-up

This starts dependencies plus the backend and worker in the background.

5. Open Paperwise

http://localhost:8000
Command What it does
make dev-status Show backend/worker process state and dependency container status.
make dev-stop Stop backend and worker started by make dev-up.
make dev-restart Restart the local development stack after changes.
make test Run the test suite from the local virtualenv.

First Run

Initial setup inside Paperwise

After the app is running, there is one more required step: configure an AI connection in the web UI. Uploads and AI-backed features depend on the model settings you save per user.

Account required Paperwise has no guest mode. Create your first account or sign in before using the app.
Where to configure models Open Settings and then Model Config after signing in.

1. Create your first account

On first launch, register a user account in the Paperwise sign-in screen. The model configuration is stored per user, so this is the account that will own your connections and task routing.

2. Add a model connection

Go to Settings > Model Config, add a connection, and choose your provider. Paperwise currently supports OpenAI, Gemini, and custom OpenAI-compatible providers.

Enter the provider API key and, if needed, a base URL. You can keep one general-purpose connection or create multiple connections for different tasks.

3. Assign task models

In the same Model Config section, choose which connection and model each task should use:

  • Metadata Extraction: required before uploads and extraction workflows are available.
  • Grounded Q&A: required if you want Ask My Docs and source-backed answers.
  • OCR: choose either LLM or Local Tesseract.

4. Pick an OCR mode

If you choose LLM for OCR, assign an OCR connection and model just like the other tasks. If you choose Local Tesseract, OCR runs locally instead of using an API model.

The support page has more guidance on OCR tradeoffs and model choices if you want to tune this further.

5. Save settings and test with a document

Save the model configuration, upload a sample PDF, and confirm that metadata extraction, OCR, or grounded Q&A behave the way you expect.

Feature Minimum setup
Document upload and extraction A working Metadata Extraction connection in Settings > Model Config.
Ask My Docs A working Grounded Q&A connection in Settings > Model Config.
OCR on scans and image PDFs Either an OCR LLM connection or Local Tesseract enabled.
Read model setup guidance Open repository

Not sure which option to choose?

Use Docker if you want the shortest path to a working self-hosted install. Use local dev mode if you plan to modify the code or debug the internals.
